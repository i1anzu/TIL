# Git Flow

**Git Flow란?** Git을 사용하여 협업할 때 브랜치를 효율적으로 관리하기 위한 약속(브랜치 전략)입니다. 2010년 Vincent Driessen이 제안한 모델로, 프로젝트의 각 단계(개발, 테스트, 배포, 버그 수정)마다 사용하는 브랜치를 명확히 나누어 관리합니다.

<**핵심 개념**>
- **Main (Master)**: 최종 제품으로 출시되는 코드가 담긴 브랜치입니다. 항상 실행 가능한 상태여야 합니다.
- **Develop**: 다음 출시를 위해 기능을 합치고 통합 테스트를 진행하는 메인 개발 브랜치입니다.
- **Release**: 배포를 준비하는 브랜치입니다. develop에서 출시 준비가 완료되면 생성하며, 최종 버그 수정 후 main과 develop 양쪽으로 합칩니다.
- **Hotfix**: 출시된 제품(main)에서 긴급하게 수정해야 할 버그가 생겼을 때 사용합니다.

<**작업 흐름 (Workflow)**>
- **기능 개발**: develop 브랜치에서 feature 브랜치를 생성하여 작업한 뒤, 완료되면 다시 develop에 합칩니다.
- **배포 준비**: 다음 버전 출시가 결정되면 develop에서 release 브랜치를 만듭니다. 여기서 마지막 점검을 합니다.
- **정식 배포**: release 내용을 main에 합치고 버전을 태그(v1.0 등)로 기록합니다. 변경 사항은 다시 develop에도 반영합니다.
- **긴급 수정**: 서비스 중 문제가 터지면 main에서 hotfix를 만들어 고친 후, 다시 main과 develop에 합칩니다.

<**장점과 단점**>
- **장점**: 역할 구분이 명확해 대규모 팀이나 정기 배포가 중요한 프로젝트에 매우 체계적입니다.
- **단점**: 구조가 복잡하여 관리가 번거롭고, 지속적으로 배포(CI/CD)되는 현대의 웹 서비스에는 GitHub Flow처럼 더 단순한 방식이 더 선호되기도 합니다.
